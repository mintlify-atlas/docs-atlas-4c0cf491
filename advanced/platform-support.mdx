## macOS Support

### System Requirements

- **macOS Version**: 10.15 (Catalina) or later
- **Architecture**: Intel x86_64, Apple Silicon (ARM64)
- **Permissions**: Accessibility permissions required

### macOS-Specific Implementation

```
internal/core/infra/platform/darwin/
├── accessibility_element_darwin.m    # AX element queries
├── accessibility_screen_darwin.m     # Screen coordinate conversion
├── eventtap_darwin.m                 # Keyboard event interception
├── overlay_darwin.m                  # Native window rendering
├── system.go                         # System info, dark mode detection
└── bridge.go                         # CGo bridge layer
```

### Coordinate System

Neru uses a **global top-left (0,0) coordinate system** for all shared logic. macOS Cocoa APIs use a bottom-left (0,0) coordinate system where Y increases upwards. The darwin platform adapter automatically inverts the Y coordinate before passing it to shared Go code.

## Cross-Platform Architecture

### The "One Rule"

<Warning>
**Non-darwin-tagged code must never import `internal/core/infra/platform/darwin`.**

This is enforced by `golangci-lint` using `depguard` to ensure platform isolation.
</Warning>

### Design Principles

1. **Shared Business Logic**: All core logic (hint generation, grid calculations, mode transitions) is written in pure Go and resides in `internal/core/domain` and `internal/app/services`.

2. **Platform Isolation**: OS-specific code is strictly isolated. Non-darwin code must never import macOS-specific packages.

3. **Ports and Adapters**: System capabilities (Accessibility, Hotkeys, Overlays) are defined as interfaces (Ports) in `internal/core/ports`. Concrete implementations (Adapters) reside in `internal/core/infra`.

4. **Build Tag Separation**: OS-specific files use Go build tags (e.g., `//go:build darwin`) to ensure they are only compiled for the target platform.

### Port Interfaces

All platform-specific capabilities are abstracted behind port interfaces:

```go
// internal/core/ports/system.go
type SystemPort interface {
    GetMainScreenBounds() (Rect, error)
    GetCursorPosition() (Point, error)
    IsDarkMode() (bool, error)
    // ...
}

// internal/core/ports/accessibility.go
type AccessibilityPort interface {
    GetClickableElements() ([]Element, error)
    ClickElement(element Element) error
    // ...
}
```

### Platform Factory

The platform factory pattern ensures clean separation:

```
internal/core/infra/platform/
├── factory.go              # Interface definition
├── factory_darwin.go       # macOS implementation
├── factory_linux.go        # Linux stub
└── factory_windows.go      # Windows stub
```

## Contributing to Platform Support

**Looking for cross-platform tasks?** Check issues labeled [`cross-platform`](https://github.com/y3owk1n/neru/issues?q=is%3Aopen+is%3Aissue+label%3Across-platform) or join our [Linux Support Discussion](https://github.com/y3owk1n/neru/discussions/559).

### How to Add a Feature

1. **Find the right adapter**: Locate the platform-specific implementation file (e.g., `internal/core/infra/platform/linux/system.go`).

2. **Replace the stub**: Most unimplemented methods return `derrors.CodeNotSupported`. Replace with a real implementation.

3. **Add tests**: Unit tests go next to the implementation. Integration tests use build tags like `//go:build integration && linux`.

4. **Run CI locally**:
   ```bash
   just lint
   just vet
   just test
   just build
   ```

See the [Architecture Documentation](https://github.com/y3owk1n/neru/blob/main/docs/ARCHITECTURE.md) for detailed contributor guidelines.

## Error Handling

When a platform-specific feature is not yet implemented, the adapter returns an error with the `CodeNotSupported` code:

```go
return derrors.New(derrors.CodeNotSupported, "feature X not yet implemented on linux")
```

Callers in the service layer use the `IsNotSupported(err)` helper to handle missing features gracefully (e.g., by logging a warning instead of returning an error to the user).
